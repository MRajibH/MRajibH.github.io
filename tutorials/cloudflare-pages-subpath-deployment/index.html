Deploying static apps under a Cloudflare Pages subpath without breaking assets, routing, or trust

Topic: Cloudflare deployment Audience: Frontend and platform engineers Theme: Path-based hosting discipline

Hosting an app at a subpath such as /resume-studio/ seems small until you discover how many parts of the stack assume root hosting. Asset URLs, canonical links, internal navigation, sitemap entries, service workers, and even client-side PDF exports can all quietly assume the app lives at /. Cloudflare Pages is excellent for static delivery, but path-based routing usually needs more deliberate work than a normal hostname deployment.

The first principle is simple: choose one routing truth and carry it through the whole stack. If the app must live at a subpath, the frontend should know that, the static assets should know that, and the platform routing layer should know that. Problems usually appear when only one of those layers is updated.

What usually has to change

  • Framework base path or asset prefix configuration.
  • Internal links, metadata, and canonical URLs.
  • Worker or proxy rules that rewrite the request path correctly.
  • Verification of generated static asset URLs after deployment.

Where teams get surprised

The homepage may load correctly while nested routes, icons, PDFs, or client-side downloads still break because one part of the app is quietly building root-relative URLs.

Why the worker layer matters

Cloudflare Pages custom domains map cleanly to hostnames, but path-based publishing often benefits from a Worker route that rewrites or proxies subpath traffic to the correct Pages project. That worker becomes the thin layer that translates user-facing URL shape into origin-facing asset delivery. It should stay small, predictable, and well documented because it becomes part of the deployment contract.

I also like to verify the deployed HTML directly after release. If the public page still references root-relative assets when it should reference subpath assets, the app is not actually deployed correctly even if the first paint looks fine. Path deployments fail in details, so the verification process needs to look at details too.

A subpath deployment is stable only when links, assets, metadata, and routing all agree about where the app lives.

Trust and crawl implications

Path-based hosting affects more than runtime behavior. Search engines, ad reviewers, and users all notice when navigation feels inconsistent or when policy pages and favicons resolve strangely. That is why I treat canonical URLs, sitemap entries, and footer navigation as part of the deployment work rather than post-launch cleanup. A technical deployment is also a publishing deployment when the site is public.

Path-based hosting is reliable when publishing details are treated as infrastructure, not decoration.
